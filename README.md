MariaDB has two sides:

***mariadb-server*** — where the files will be stored, and the server waits for communication by default via port 3306. The data is physically saved on disk at /var/lib/mysql/. However in Docker, this folder is temporary and wiped on restart so you must use a volume to map this folder to the host machine to keep the data persistent.

***mariadb-client*** — a command-line tool responsible for talking to the server and sending SQL queries. Your application (website, backend) also acts as a client, communicating with the server the same way through port 3306.


---------------


***Nginx*** is a web server — its job is to receive requests from browsers and respond to them, either by serving files directly or forwarding requests to another service.
In this project, Nginx is the entry point of the project. Every browser request passes through it first via port 443 (HTTPS).
It does two things:

Static files (images, CSS) → served directly from /var/www/html/ without any processing
PHP files → forwarded to the WordPress container on port 9000 because Nginx can't execute PHP itself

All communication is encrypted using a TLS 1.2 certificate generated by OpenSSL, stored in /etc/nginx/ssl/.


***FastCGI*** is a protocol — a set of rules that defines how Nginx and PHP-FPM communicate with each other. It packages all request information (which file to run, what parameters were sent, browser IP, etc.) and passes it to PHP-FPM, then brings the result back to Nginx.
In this project, FastCGI is the bridge between Nginx and WordPress. Without it they have no way to talk — Nginx doesn't speak PHP and PHP-FPM doesn't handle HTTP requests.
When Nginx receives a .php request it packages everything into FastCGI variables and sends them to wordpress:9000. PHP-FPM unpacks them, executes the PHP file, and returns the HTML result back through the same bridge.

---------------


***WordPress*** is a Content Management System (CMS) built in PHP — it provides a ready-made website with an admin panel, database structure, and plugin/theme system so you don't build everything from scratch.
In this project, WordPress is a separate Docker container that acts as the PHP processor. 
It sits between Nginx and MariaDB it receives PHP requests from Nginx, executes them, talks to MariaDB if needed, and returns the result.

It runs PHP-FPM on port 9000 — always listening and waiting for Nginx to forward PHP requests.
The WordPress files live in /var/www/html/ which is a shared volume between Nginx and WordPress — Nginx needs the files to serve static assets, WordPress needs them to execute PHP.
WordPress runs in the foreground (-F) to keep the Docker container alive.

DOCKER:

Namespaces — A Linux kernel feature that isolates what a container can see (processes, network, filesystem, hostname).

Cgroups — A Linux kernel feature that isolates what a container can use (CPU, memory, disk, network ).

runc — A low-level CLI tool that reads the container spec and makes direct syscalls to the Linux kernel to create the container.

Dockerfile
A plain text file containing a set of instructions that tells Docker how to build an image. Each instruction represents a step like choosing a base, installing software, or copying files. It's the blueprint.


Docker Image
A read-only stack of layers stored on disk, produced by building a Dockerfile. Each layer is a diff (only changed files) from the previous instruction. When run, it becomes a container. It's the built result of the blueprint. And the responsible for thr creation is "BuildKit"




Docker Container:
A container is a running isolated process built from an image, created by dockerd → containerd → runc → Linux kernel, isolated via Namespaces and Cgroups, with its own writable layer on top of the image's read-only layers.


Docker compose:
Docker Compose is a tool for defining and managing multi-container Docker applications using a single YAML file, allowing you to configure, connect, and run all your containers together with one command.

# Inception

## Description

**Inception** is a system administration project from the 42 curriculum. Its goal is to broaden knowledge of system administration using Docker by building a small but complete infrastructure composed of multiple services, each running in its own dedicated container.

The infrastructure consists of three core services:

- **NGINX** — the entry point of the project. It listens exclusively on port 443 (HTTPS) and handles all incoming browser requests. Static files are served directly from `/var/www/html/`. PHP requests are forwarded to the WordPress container via the **FastCGI** protocol on port 9000. All traffic is encrypted using a **TLS 1.2** certificate generated with OpenSSL, stored in `/etc/nginx/ssl/`.

- **WordPress** — a PHP-based Content Management System running in its own container. It acts as the PHP processor between NGINX and MariaDB: it receives PHP requests forwarded by NGINX via FastCGI, executes them, queries MariaDB when needed, and returns the result. It runs **PHP-FPM** on port 9000. WordPress files live in `/var/www/html/`, a shared Docker volume between NGINX and WordPress.

- **MariaDB** — the relational database that stores all application data (users, posts, settings, etc.). It listens on port 3306. Its data is stored at `/var/lib/mysql/` inside the container, which is mapped to a persistent Docker volume on the host to survive container restarts.

All three services communicate through a dedicated **Docker network** and are orchestrated using **Docker Compose**.

---

### Docker — How It Works

Docker uses two core Linux kernel features to create isolated containers:

- **Namespaces** — isolate what a container can *see* (processes, network, filesystem, hostname)
- **Cgroups** — isolate what a container can *use* (CPU, memory, disk, network bandwidth)

Containers are created through a chain: `dockerd → containerd → runc → Linux kernel`.

A **Dockerfile** is the blueprint — a set of instructions that defines how to build a Docker **image**. An image is a read-only stack of layers stored on disk, where each layer represents a diff from the previous instruction. When an image is run, it becomes a **container** — an isolated, running process with its own writable layer on top.

**Docker Compose** is a tool for defining and managing multi-container applications through a single YAML file, allowing all services to be configured, connected, and started with a single command.

### Design Choices & Comparisons

#### Virtual Machines vs Docker

| | Virtual Machines | Docker |
|---|---|---|
| **Isolation** | Full OS per VM, hardware-level isolation via hypervisor | Process-level isolation via namespaces and cgroups |
| **Boot time** | Minutes | Milliseconds |
| **Resource usage** | Heavy — each VM runs a full OS kernel | Lightweight — containers share the host kernel |
| **Portability** | Large VM images, slower to distribute | Small layered images, fast to pull and push |
| **Use case** | Full OS isolation, different kernels | Microservices, fast deployment, dev/prod parity |

In this project, Docker is the right choice: we want lightweight, fast-to-build, interconnected services without the overhead of running multiple full operating systems.

#### Secrets vs Environment Variables

| | Environment Variables | Docker Secrets |
|---|---|---|
| **Storage** | Stored in plain text in `.env` files or the environment | Stored encrypted, mounted as in-memory files at `/run/secrets/` |
| **Visibility** | Visible via `docker inspect`, process lists, logs | Only accessible inside the container that needs them |
| **Security** | Suitable for non-sensitive config (ports, usernames) | Required for sensitive data (passwords, API keys, certificates) |
| **Use case** | General configuration | Credentials and secrets in production |

In this project, Docker Secrets are used for database passwords and WordPress credentials to avoid exposing sensitive data in environment variables or image layers.

#### Docker Network vs Host Network

| | Docker Network (bridge) | Host Network |
|---|---|---|
| **Isolation** | Containers get their own virtual network, isolated from the host | Container shares the host's network namespace directly |
| **Communication** | Containers communicate using service names as DNS | Containers use `localhost` like any host process |
| **Security** | Better isolation; ports must be explicitly exposed | No isolation; all ports are directly accessible on the host |
| **Use case** | Multi-container apps that need internal communication | High-performance use cases, or when direct host access is needed |

In this project, a custom Docker bridge network is used so that NGINX, WordPress, and MariaDB can communicate by service name (e.g., `wordpress:9000`, `mariadb:3306`) without exposing internal ports to the outside world.

#### Docker Volumes vs Bind Mounts

| | Docker Volumes | Bind Mounts |
|---|---|---|
| **Managed by** | Docker (stored in `/var/lib/docker/volumes/`) | The user (maps to any host path) |
| **Portability** | Fully managed, easy to back up and migrate | Depends on the host filesystem path existing |
| **Performance** | Optimized by Docker | Depends on host filesystem |
| **Use case** | Persistent data (databases, uploads) | Development (live code reloading, config files) |

In this project, Docker Volumes are used for:
- **MariaDB data** (`/var/lib/mysql/`) — to persist the database across container restarts
- **WordPress files** (`/var/www/html/`) — shared between NGINX and WordPress so both can access the same files

## Instructions

### Prerequisites

- Docker Engine
- Docker Compose
- `make`

### Setup

1. Clone the repository:
   ```bash
   git clone <your-repo-url>
   cd inception
   ```

2. Create a `.env` file at the root with the required environment variables (refer to `.env.example` if provided).

3. Build and start all containers:
   ```bash
   make
   ```
   This will call `docker compose up --build -d` under the hood.

4. To stop and remove all containers:
   ```bash
   make down
   ```

5. To remove all containers, volumes, and built images (full clean):
   ```bash
   make fclean
   ```

### Access

Once running, the site is available at:
```
https://localhost
```
> Accept the self-signed TLS certificate warning in your browser (expected for local development).
